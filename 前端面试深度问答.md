# 一、5年前端面试的核心定位

5年前端，面试官一般不会只问“会不会写页面”，而是重点看这几件事：

1. **基础是否扎实**：JS、浏览器、网络、CSS、异步、模块化。
2. **框架是否理解原理**：React / Vue 的响应式、渲染、状态管理、性能优化。
3. **项目是否有复杂度**：权限、动态表单、通用表格、请求封装、性能治理、工程化。
4. **能不能独立负责模块**：需求拆解、方案设计、踩坑复盘、推动落地。
5. **有没有工程判断力**：什么场景用什么方案，为什么不用另一个方案。
6. **是否具备协作和带人能力**：Code Review、规范制定、组件沉淀、排查线上问题。

所以回答时不要只说“我会”，而要说：

> 我在什么业务场景下遇到什么问题，为什么选这个方案，具体怎么实现，效果如何，有什么权衡，后续还能怎么优化。

# 二、JS 深度高频问答

## 1. 说一下原型链

**面试官问：** JS 原型链是什么？

**普通回答：** 每个对象都有原型，对象访问属性时会沿着原型链查找。

**5年经验回答：**

JS 的继承主要基于原型链。每个对象都有一个内部原型指向，通常可以通过 `__proto__` 访问；函数有 `prototype` 属性，用于作为通过 `new` 创建出来的实例对象的原型。

当访问一个对象属性时，JS 引擎会先在对象自身查找，如果找不到，就去它的原型对象上找，再继续向上，直到 `Object.prototype`，最后到 `null`。这条链就是原型链。

**可以继续补一句：**

在实际项目中，我更关注原型链带来的两个问题：一是方法复用，比如类实例方法挂在 prototype 上避免每个实例重复创建；二是原型污染风险，比如合并对象时不能无脑递归处理 `__proto__`、`constructor`、`prototype` 这些 key。

**追问：`new` 做了什么？**

`new` 大致做了四件事：

```js
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```

核心是：创建对象、链接原型、绑定 this、返回对象。

## 2. 闭包是什么？有什么应用和风险？

**标准回答：**

闭包是指函数可以访问其词法作用域外层变量，即使外层函数已经执行完毕，这些变量仍然不会被垃圾回收。

**项目化回答：**

闭包本质上是函数和其词法环境的组合。实际开发里，闭包常用于：

1. 数据私有化。
2. 防抖节流。
3. 缓存结果。
4. 保存异步任务中的上下文。
5. React Hooks 内部也大量依赖闭包特性。

**风险：**

闭包如果长期引用大对象、DOM 节点、定时器变量，可能导致内存无法释放。比如组件卸载后没有清理定时器、事件监听，闭包还引用旧状态，就可能造成内存泄漏。

**追问：React 里闭包有什么坑？**

典型问题是 stale closure，旧闭包拿到的是旧 state。

```js
function App() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const timer = setInterval(() => {
      console.log(count)
    }, 1000)

    return () => clearInterval(timer)
  }, [])
}
```

这里永远打印初始 count。解决方式可以用依赖数组、函数式更新或 `useRef` 保存最新值。

## 3. 事件循环怎么讲？

**面试官问：** 宏任务和微任务区别？

**回答框架：**

JS 是单线程执行的，但浏览器环境提供了事件循环机制。同步代码先进入调用栈执行，异步任务根据类型进入不同队列。当前宏任务执行完后，会清空所有微任务，然后再进入下一个宏任务。

常见宏任务：

```txt
script
setTimeout
setInterval
setImmediate(Node)
I/O
UI rendering
```

常见微任务：

```txt
Promise.then
queueMicrotask
MutationObserver
process.nextTick(Node)
```

**输出题回答口诀：**

先执行同步代码，再执行微任务，再执行下一个宏任务。

```js
console.log(1)

setTimeout(() => {
  console.log(2)
}, 0)

Promise.resolve().then(() => {
  console.log(3)
})

console.log(4)
```

输出：

```txt
1
4
3
2
```

**5年经验补充：**

在实际项目里，我会关注事件循环对性能和交互的影响。比如大量同步计算会阻塞主线程，导致页面卡顿；大量微任务连续执行，也可能延迟渲染。对于大任务可以拆分成分片执行，或者使用 Web Worker。

## 4. Promise 原理怎么说？

**核心回答：**

Promise 是一种异步状态管理机制，有三种状态：pending、fulfilled、rejected。状态一旦改变就不可逆。`then` 会注册回调，并且返回一个新的 Promise，所以可以链式调用。

**手写简化版思路：**

```js
class MyPromise {
  constructor(executor) {
    this.state = 'pending'
    this.value = undefined
    this.reason = undefined
    this.onFulfilledCallbacks = []
    this.onRejectedCallbacks = []

    const resolve = (value) => {
      if (this.state !== 'pending') return
      this.state = 'fulfilled'
      this.value = value
      this.onFulfilledCallbacks.forEach(fn => fn())
    }

    const reject = (reason) => {
      if (this.state !== 'pending') return
      this.state = 'rejected'
      this.reason = reason
      this.onRejectedCallbacks.forEach(fn => fn())
    }

    try {
      executor(resolve, reject)
    } catch (err) {
      reject(err)
    }
  }

  then(onFulfilled, onRejected) {
    return new MyPromise((resolve, reject) => {
      if (this.state === 'fulfilled') {
        queueMicrotask(() => {
          try {
            const result = onFulfilled(this.value)
            resolve(result)
          } catch (err) {
            reject(err)
          }
        })
      }

      if (this.state === 'rejected') {
        queueMicrotask(() => {
          try {
            const result = onRejected(this.reason)
            resolve(result)
          } catch (err) {
            reject(err)
          }
        })
      }

      if (this.state === 'pending') {
        this.onFulfilledCallbacks.push(() => {
          queueMicrotask(() => {
            try {
              const result = onFulfilled(this.value)
              resolve(result)
            } catch (err) {
              reject(err)
            }
          })
        })
      }
    })
  }
}
```

**追问：async/await 原理？**

`async/await` 本质上是 Generator + Promise 的语法糖。`async` 函数返回 Promise，`await` 会暂停后续代码执行，等待 Promise 状态改变后，再把后续逻辑放入微任务继续执行。

## 5. 防抖和节流怎么讲？

**防抖 debounce：**

连续触发时，只在最后一次触发后执行。

适合：

```txt
搜索框输入联想
窗口 resize
表单校验
```

```js
function debounce(fn, delay = 300) {
  let timer = null

  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
    }, delay)
  }
}
```

**节流 throttle：**

一段时间内最多执行一次。

适合：

```txt
滚动监听
拖拽
按钮防连点
滚动加载
```

```js
function throttle(fn, delay = 300) {
  let last = 0

  return function (...args) {
    const now = Date.now()
    if (now - last >= delay) {
      last = now
      fn.apply(this, args)
    }
  }
}
```

**加分回答：**

项目中不是所有输入都适合防抖，比如金额输入、权限搜索、远程下拉，如果用户期望即时反馈，可以用防抖加 loading 状态；按钮提交则要结合前端禁用、后端幂等、请求去重共同处理。

# 三、React 高频问答

## 1. React 的渲染流程怎么讲？

React 官方把 UI 更新过程拆成 trigger、render、commit。render 阶段本质是调用组件函数计算新的 UI 描述，commit 阶段才把结果提交到 DOM。React 官方文档也明确说明，rendering 是 React 调用组件以确定屏幕上要显示什么。([React][1])

**面试回答：**

React 更新大致分为三步：

1. 触发更新：state、props、context 变化。
2. render 阶段：执行组件函数，生成新的虚拟 DOM。
3. diff / reconciliation：比较新旧虚拟 DOM。
4. commit 阶段：把变更提交到真实 DOM，并执行 layout effect、effect 等副作用。

**加分点：**

React 的 render 阶段应该保持纯净，不要直接操作 DOM、发请求或修改外部变量。副作用应该放到 `useEffect` 或 `useLayoutEffect` 中。

---

## 2. React 为什么需要 key？

**回答：**

key 用于帮助 React 在列表 diff 时识别节点身份。没有稳定 key 时，React 只能按照位置复用节点，可能导致错误复用、状态错乱、性能变差。

**不要只说不能用 index，要补场景：**

如果列表只是静态展示，不增删改排序，用 index 问题不大。但如果列表会插入、删除、排序、过滤，用 index 会导致组件状态错位，比如输入框内容错乱、动画异常。

---

## 3. useMemo 和 useCallback 怎么用？有什么坑？

`useMemo` 用于在依赖不变时缓存计算结果，React 官方文档也把它定义为在重新渲染之间缓存计算结果的 Hook。([React][2])

**回答：**

`useMemo` 缓存值，`useCallback` 缓存函数引用。

适合场景：

```txt
复杂计算
大列表过滤
传给 memo 子组件的函数
依赖引用稳定性的场景
```

**不要滥用：**

如果计算很轻，或者组件本身不会频繁渲染，滥用 `useMemo` / `useCallback` 反而增加心智负担和依赖维护成本。

**追问：为什么用了 useCallback 子组件还是渲染？**

因为子组件默认只要父组件渲染就会重新执行。需要配合 `React.memo`，并且 props 引用要稳定。

---

## 4. useEffect 和 useLayoutEffect 区别？

**回答：**

`useEffect` 在浏览器完成绘制后异步执行，不会阻塞页面渲染。
`useLayoutEffect` 在 DOM 更新后、浏览器绘制前同步执行，会阻塞渲染。

**使用场景：**

`useEffect` 适合请求、订阅、日志、定时器。
`useLayoutEffect` 适合需要在用户看到页面前完成 DOM 测量或样式修正的场景，比如弹窗定位、滚动位置恢复。

**项目经验回答：**

我一般默认用 `useEffect`，只有遇到明显闪烁、需要同步测量 DOM 尺寸时才使用 `useLayoutEffect`，因为它会阻塞绘制，滥用会影响性能。

---

## 5. React 状态管理怎么选？

**回答框架：**

状态可以按作用域分层：

```txt
组件内部状态：useState / useReducer
跨组件局部共享：Context
中大型业务全局状态：Redux Toolkit / Zustand / Pinia 类似思路
服务端缓存状态：React Query / SWR
URL 状态：query、params
持久化状态：localStorage / IndexedDB
```

**5年经验回答：**

我不会把所有状态都塞进 Redux。比如表单临时输入、弹窗开关、组件 UI 状态适合放组件内部；用户信息、权限、主题、布局配置适合全局；列表查询参数可以放 URL，方便刷新和分享；接口数据缓存可以交给 React Query 这类工具。

# 四、Vue 高频问答

## 1. Vue 响应式原理怎么讲？

Vue 官方文档说明，Vue 的响应式系统会把普通 JS 对象深度转换为 reactive proxies，也就是基于 Proxy 实现依赖追踪和更新触发。([Vue.js][3])

**面试回答：**

Vue 3 使用 Proxy 对对象进行代理。当组件渲染时访问响应式数据，会触发 getter 收集依赖；当数据变化时，会触发 setter 通知相关 effect 重新执行，从而更新视图。

核心过程：

```txt
reactive/ref 创建响应式数据
组件渲染读取数据
track 收集依赖
数据修改
trigger 触发更新
scheduler 调度渲染
```

---

## 2. computed 和 watch 区别？

Vue 官方文档说明，computed 会基于响应式依赖进行缓存，只有依赖变化时才会重新计算。([Vue.js][4])

**回答：**

`computed` 适合从已有状态推导新状态，有缓存，强调“结果”。
`watch` 适合监听数据变化后执行副作用，强调“动作”。

比如：

```txt
computed：总价、筛选后的列表、按钮是否可点击
watch：监听路由变化发请求、监听关键词远程搜索、监听表单值回显
```

**追问：watchEffect 呢？**

`watchEffect` 会自动收集回调中使用到的响应式依赖，适合依赖简单、想快速响应的场景；`watch` 更适合明确监听某个数据，并拿到 oldValue 和 newValue。

---

## 3. Vue nextTick 是什么？

**回答：**

Vue 更新 DOM 是异步批处理的。数据变化后，DOM 不会立即同步更新，而是进入更新队列。`nextTick` 可以在 DOM 更新完成后执行回调。

**项目场景：**

比如弹窗打开后需要获取元素高度、表格数据更新后滚动到底部、表单项动态显示后调用校验，这些都可能需要 `nextTick`。

---

## 4. Vue 组件通信方式有哪些？

**回答：**

```txt
父传子：props
子传父：emit
多层传递：provide/inject
兄弟组件：状态提升、事件总线、store
全局状态：Pinia / Vuex
路由参数：route params/query
```

**5年经验回答：**

我一般不会滥用全局状态。组件内聚的状态就放组件内部；页面级状态可以放页面组件；多个页面共享的用户信息、权限、主题配置再放 Pinia。否则状态来源太多，后期很难维护。

# 五、工程化高频问答

## 1. Vite 为什么快？

**回答：**

Vite 快主要体现在开发阶段。它利用浏览器原生 ES Module，启动时不需要像 Webpack 那样先把整个项目打包一遍，而是按需加载模块。依赖预构建使用 esbuild，速度很快。

**追问：生产环境为什么还是 Rollup？**

因为生产环境更关注构建产物优化，比如 tree shaking、代码分割、chunk 优化、资源处理，Rollup 在库构建和产物优化方面更成熟。

---

## 2. Webpack 优化你做过哪些？

**回答：**

常见优化分两类：构建速度优化和产物体积优化。

构建速度：

```txt
缓存 cache
缩小 loader 范围 include/exclude
thread-loader
esbuild-loader / swc-loader
DLL 或依赖预构建
```

产物体积：

```txt
tree shaking
splitChunks
路由懒加载
组件懒加载
按需引入
图片压缩
gzip / brotli
externals
```

**项目化回答：**

我做过一次后台系统首屏优化，先用 bundle analyzer 找出体积大的依赖，比如富文本、图表库、Excel 导出库，然后改成按需加载。路由也做了懒加载，最终首屏 JS 体积下降，页面首次加载明显变快。

---

## 3. 前端项目怎么做环境管理？

**回答：**

一般通过 `.env` 文件区分环境：

```txt
.env.development
.env.test
.env.staging
.env.production
```

配置内容包括：

```txt
API_BASE_URL
APP_ENV
CDN_URL
SENTRY_DSN
ENABLE_MOCK
```

**注意点：**

前端环境变量会被打包进静态资源，不能放真正的密钥。比如数据库密码、后端私钥、云服务 Secret Key 都不能放前端。

---

## 4. 你们怎么做代码规范？

**回答：**

我一般从四层做：

```txt
格式规范：Prettier
语法规范：ESLint
类型规范：TypeScript
提交规范：Husky + lint-staged + commitlint
```

**5年经验加分：**

规范不是越多越好，关键是能自动化。凡是能通过工具修复的，就不要靠人肉 Code Review。Code Review 应该重点看架构、边界条件、异常处理、可维护性，而不是纠结缩进和分号。

# 六、浏览器和网络高频问答

## 1. 从输入 URL 到页面展示发生了什么？

**回答框架：**

```txt
URL 解析
DNS 查询
TCP 连接
TLS 握手
发送 HTTP 请求
服务端处理并返回响应
浏览器解析 HTML
构建 DOM 树
解析 CSS 构建 CSSOM
合成 Render Tree
布局 Layout
绘制 Paint
合成 Composite
加载 JS、图片、字体等资源
```

**加分点：**

如果是 SPA，还会下载 JS bundle，执行框架初始化，挂载根组件，然后请求接口渲染数据。

---

## 2. 浏览器缓存怎么讲？

**回答：**

浏览器缓存分为强缓存和协商缓存。

强缓存：

```txt
Cache-Control
Expires
```

协商缓存：

```txt
ETag / If-None-Match
Last-Modified / If-Modified-Since
```

**项目策略：**

HTML 通常不强缓存，保证能拿到最新入口文件。
JS/CSS 使用 hash 文件名，可以设置长缓存。
图片和字体可以根据更新频率设置较长缓存。

---

## 3. CORS 是什么？

**回答：**

CORS 是浏览器的同源策略限制下，服务端声明允许跨域访问的一套机制。关键响应头包括：

```txt
Access-Control-Allow-Origin
Access-Control-Allow-Methods
Access-Control-Allow-Headers
Access-Control-Allow-Credentials
```

**追问：为什么 Postman 不跨域？**

跨域是浏览器安全策略，不是 HTTP 本身限制。Postman、curl 不受浏览器同源策略限制。

---

## 4. Cookie、Session、Token、JWT 区别？

**回答：**

Cookie 是浏览器存储机制，会自动随请求发送。
Session 是服务端保存登录状态，客户端通常保存 sessionId。
Token 是一种认证凭证，前端主动携带。
JWT 是一种自包含 token，里面可以包含用户身份、过期时间等信息。

**安全补充：**

Cookie 要关注 `HttpOnly`、`Secure`、`SameSite`。
Token 要关注过期、刷新、吊销、存储位置和 XSS 风险。

---

# 七、安全高频问答

## 1. XSS 是什么？怎么防？

OWASP 对 XSS 的描述是，攻击者可以注入内容，造成账号冒用、观察用户行为、加载外部内容、窃取敏感数据等严重后果。([OWASP Cheat Sheet Series][5])

**回答：**

XSS 是跨站脚本攻击，本质是攻击者把恶意脚本注入到页面中执行。

类型：

```txt
存储型 XSS
反射型 XSS
DOM 型 XSS
```

防御：

```txt
输入校验
输出转义
避免 v-html / dangerouslySetInnerHTML
富文本白名单过滤
CSP 内容安全策略
Cookie 设置 HttpOnly
URL 协议白名单
```

**项目经验回答：**

如果业务必须支持富文本，我不会简单信任前端过滤，而是前后端都做白名单清洗。前端展示时也避免直接插入未经处理的 HTML。

---

## 2. CSRF 是什么？怎么防？

**回答：**

CSRF 是跨站请求伪造。用户登录 A 网站后，攻击者诱导用户访问 B 页面，B 页面偷偷向 A 发请求，因为浏览器会自动带上 A 的 Cookie，所以可能造成越权操作。

防御：

```txt
SameSite Cookie
CSRF Token
Referer / Origin 校验
关键操作二次确认
不用 GET 做修改操作
```

---

## 3. 前端如何防止接口重复提交？

**回答：**

前端层面：

```txt
按钮 loading 禁用
请求去重
防抖节流
AbortController 取消重复请求
```

后端层面：

```txt
幂等 key
唯一索引
状态机控制
分布式锁
```

**加分回答：**

真正可靠的防重复提交不能只靠前端。前端只能减少误操作，最终要靠后端幂等保证。

# 八、性能优化高频问答

## 1. 你怎么做性能优化？

MDN 对 Web Performance 的定义包括加载时间、可交互性、响应性和交互流畅度，不只是“页面打开快”。([MDN文档][6])

**回答框架：**

我会按链路拆：

```txt
加载性能
渲染性能
交互性能
网络性能
资源体积
缓存策略
监控体系
```

具体措施：

```txt
路由懒加载
组件懒加载
图片压缩和懒加载
CDN
gzip / brotli
代码分割
tree shaking
虚拟列表
分页加载
骨架屏
接口缓存
减少重排重绘
长任务拆分
Web Worker
```

MDN 也把懒加载定义为识别非阻塞、非关键资源，并在需要时再加载，从而缩短关键渲染路径。([MDN文档][7])

---

## 2. 首屏加载慢怎么排查？

**回答：**

我会按这个顺序排查：

```txt
Network 看资源体积和接口耗时
Performance 看主线程长任务
Lighthouse 看 LCP / INP / CLS
Coverage 看无用 JS/CSS
Bundle Analyzer 看大依赖
接口日志看后端耗时
```

优化方向：

```txt
减少首屏 JS
拆分非首屏模块
优化首屏接口
SSR / SSG
图片使用 WebP/AVIF
关键 CSS 内联
预加载关键资源
延迟加载非关键资源
```

---

## 3. 大列表怎么优化？

**回答：**

大列表卡顿通常有两个原因：DOM 节点太多、渲染计算太重。

方案：

```txt
虚拟列表
分页加载
懒加载
减少响应式数据层级
列表项 memo
避免内联复杂计算
滚动事件节流
```

**项目化回答：**

比如一个表格有几千条数据，如果全部渲染，DOM 数量会非常大。我会使用虚拟滚动，只渲染可视区域附近的数据，同时固定行高或提前测量动态行高。

## 2. type 和 interface 区别？

**回答：**

`interface` 更适合描述对象结构，可以重复声明并合并。
`type` 更灵活，可以定义联合类型、交叉类型、条件类型、工具类型。

```ts
interface User {
  id: number
  name: string
}

type Status = 'pending' | 'success' | 'error'
```

**项目经验：**

我一般约定：对象模型优先用 interface，复杂类型组合用 type。但团队一致性比个人偏好更重要。

---

## 3. 泛型怎么讲？

**回答：**

泛型就是把类型当参数传入，让函数、组件、类在保持类型安全的同时具备复用能力。

```ts
function getValue<T>(value: T): T {
  return value
}
```

实际场景：

```ts
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

interface User {
  id: number
  name: string
}

const res: ApiResponse<User> = {
  code: 0,
  message: 'ok',
  data: {
    id: 1,
    name: 'Tom'
  }
}
```

# 十、项目深挖：后台管理系统 / SaaS 平台

## 1. 权限系统怎么设计？

**面试官问：** 你项目里的权限怎么做？

**普通回答：** 后端返回菜单，前端动态生成路由。

**5年经验回答：**

我们的权限分为四层：

```txt
菜单权限
路由权限
按钮权限
接口权限
```

登录后，后端返回用户信息、角色、权限码和菜单树。前端根据菜单树生成动态路由，同时把权限码存到全局状态中。页面按钮通过权限指令或权限组件控制显示。接口权限以后端校验为准，前端只做体验层面的隐藏。

**追问：刷新后动态路由丢失怎么办？**

刷新后内存状态丢失，所以需要重新获取用户权限并重新注入动态路由。一般在路由守卫里判断是否已经加载过权限，如果没有，就请求权限接口、生成路由、再放行。

**追问：为什么前端按钮隐藏不等于安全？**

因为用户可以绕过前端直接调接口，所以真正的权限必须在后端校验。前端权限只是提升用户体验，减少误操作入口。

---

## 2. 动态表单怎么设计？

**回答：**

动态表单核心是 schema 驱动。每个字段用配置描述：

```ts
interface FormSchema {
  field: string
  label: string
  component: 'Input' | 'Select' | 'DatePicker' | 'Switch'
  required?: boolean
  options?: Array<{ label: string; value: string | number }>
  visible?: (formData: any) => boolean
  disabled?: (formData: any) => boolean
  rules?: any[]
}
```

实现重点：

```txt
字段渲染
默认值
校验规则
字段联动
异步 options
数据回显
提交转换
权限控制
```

**项目化话术：**

比如审批配置页面，不同业务类型对应不同字段。如果每个页面单独写表单，会有大量重复代码。我把字段抽象成 schema，通过配置渲染组件，并支持字段联动和异步数据源。这样新增业务时主要改配置，不需要重复写页面。

---

## 3. 通用表格怎么设计？

**回答：**

通用表格一般抽象这些能力：

```txt
查询表单
分页
排序
筛选
列配置
操作列
权限按钮
多选
导出
自定义渲染
状态持久化
```

列配置示例：

```ts
interface TableColumn {
  prop: string
  label: string
  width?: number
  formatter?: (row: any) => string
  render?: (row: any) => React.ReactNode
  permission?: string
}
```

**追问：通用组件会不会过度封装？**

会，所以我会控制边界。通用表格只封装稳定共性，比如分页、查询、loading、权限按钮。对于高度定制的复杂页面，保留插槽或 render 函数，不强行把所有业务都塞进配置。

---

## 4. 请求封装怎么设计？

**回答：**

请求封装一般包括：

```txt
baseURL
timeout
请求头 token
统一错误处理
登录过期处理
token 刷新
重复请求取消
loading 合并
接口重试
文件上传下载
```

**token 刷新怎么做？**

当接口返回 token 过期时，如果当前没有刷新任务，就发起刷新 token 请求；如果已经在刷新，其他请求进入队列等待。刷新成功后，重放队列请求；刷新失败则清理登录状态并跳转登录页。

**加分回答：**

要避免多个接口同时 401 导致重复刷新 token。这里需要一个 `isRefreshing` 状态和请求队列。

# 十一、面试官最爱追问的“项目难点”

## 1. 你项目里最复杂的模块是什么？

**推荐回答结构：**

```txt

实现：关键技术点
结果：带来了什么收益

```

**示例：**

我负责过后台系统的动态表单和通用表格模块。复杂点在于不同业务线字段差异大，而且存在字段联动、异步选项、权限控制和回显转换。最开始每个页面单独写，重复代码很多，维护成本高。

后来我把表单抽象为 schema 配置，通过字段类型映射组件，通过 visible、disabled、rules 实现联动和校验。表格侧也抽象了查询、分页、排序、操作列和权限按钮。这样新增一个普通 CRUD 页面时，大部分逻辑只需要写配置，开发效率提升明显。

但我也做了边界控制，没有把所有复杂业务都配置化。对于强定制页面，保留插槽和自定义 render，避免过度封装。

## 2. 你做过哪些性能优化？

**回答模板：**

我做性能优化一般不会一上来就凭感觉改，而是先量化问题。

第一步，用 Chrome Performance、Lighthouse、Network 和打包分析工具定位瓶颈。
第二步，区分是加载慢、接口慢、渲染慢还是交互卡顿。
第三步，再针对性优化。

比如某个后台首页首屏慢，我排查后发现主要有三个问题：首屏 bundle 太大、图表库一次性加载、多个接口串行请求。后来我做了路由懒加载、图表组件异步加载、接口并行和缓存，同时拆掉了一些无用依赖。优化后首屏加载速度明显改善。

## 3. 你怎么处理线上 bug？

**回答模板：**

确认影响范围
快速止血
复现问题
定位原因
修复上线
补充监控和测试
复盘避免再次发生

**示例回答：**

有一次线上出现部分用户白屏。我先通过监控确认是某个版本发布后错误率上升，然后看 Sentry 报错定位到一个接口字段为空导致解构失败。短期先回滚版本止血，随后修复代码，给关键字段增加空值兜底，并补了单元测试和接口 mock 场景。
