## 1. 项目概述

### 1.1 背景与目标

本项目是一个可视化表单设计器，支持通过拖拽方式搭建表单（行/列/字段三级嵌套布局），并具备：

- 按角色（Role）控制设计权限的 RBAC 能力
- Schema 驱动的设计态渲染与运行态（预览/填写）渲染分离
- JSON / YAML 双格式的表单 Schema 导入导出
- 基于浏览器 File System Access API 的本地文件读写（兼容降级为下载）

### 1.2 核心用户角色

| 角色 code | 说明 |
|---|---|
| `superadmin` | 超级管理员，全部权限 |
| `admin` | 管理员，全部权限 |
| `ops` | 运营，全部权限 |
| `hr` | 人力，可设计字段但不可调整布局（Row/Col） |
| `sales` | 销售，仅可预览表单 |
| `investigator` | 调查员，可预览并导出填写结果（values） |
| `user` | 普通用户，可预览、可导入导出 Schema，不可编辑 |

## 2. 系统架构

### 2.1 技术栈说明

| 层 | 技术 | 说明 |
|---|---|---|
| 构建 | Vite 7 + @vitejs/plugin-react | 开发/构建工具链 |
| UI 框架 | React 19 | 函数组件 + Hooks |
| 类型系统 | TypeScript 5.9 | 严格的 Schema 联合类型建模 |
| 样式 | Tailwind CSS 3 | 原子化 CSS，`col-span-*` 动态类已通过 safelist 处理 |
| 拖拽 | @dnd-kit/core + @dnd-kit/sortable | 行排序、列内字段排序、跨列拖拽 |
| 序列化 | js-yaml 4 | Schema 与 YAML 互转（safe schema，无 RCE 风险） |
| 状态管理 | React Context + useReducer | 无第三方状态库 |

### 2.2 模块划分

```
src/
├── app/                    # 应用入口（App.tsx）
├── auth/                   # 角色与权限（RoleContext.tsx / permissions.ts）
├── schema/                 # Schema 类型定义、默认值、校验器
│   ├── types.ts
│   ├── defaults.ts
│   └── validators.ts
├── state/                  # 设计器全局状态（Context + Reducer）
│   ├── designerContext.tsx
│   └── designerReducer.ts
├── designer/                # 设计态 UI
│   ├── FormDesigner.tsx     # 三栏布局容器
│   └── components/
│       ├── ComponentPalette.tsx  # 左侧组件库
│       ├── Canvas.tsx            # 中间画布（拖拽承载区）
│       ├── RowCard.tsx
│       ├── FieldCard.tsx
│       ├── PropertyEditor.tsx    # 右侧属性面板
│       └── TopBar.tsx            # 顶部工具栏（模式切换/导入导出）
├── renderer/                # 运行态渲染
│   ├── FormRenderer.tsx     # 只读渲染（当前疑似未被引用）
│   ├── fieldRenderers.tsx
│   └── FormRuntime.tsx      # 可交互填写渲染（预览模式实际使用）
└── utils/                   # 通用工具（deepClone / id / classnames / fileIO）
```

### 2.3 数据流

```
用户操作（拖拽/点击/输入）
      │
      ▼
dispatch(DesignerAction)
      │
      ▼
designerReducer（deepClone schema → 修改 → 返回新 state）
      │
      ▼
DesignerContext（state, dispatch）
      │
      ├─▶ Canvas / RowCard / FieldCard（设计态渲染）
      ├─▶ PropertyEditor（属性编辑，回写 dispatch）
      ├─▶ FormRuntime（预览态渲染，独立 local state 承载填写值）
      └─▶ TopBar（Schema 序列化 / 导入导出 / 校验状态展示）
```

> 状态管理特点：单一 `DesignerState`（schema + selected + mode），所有变更通过 `structuredClone` 深拷贝后修改，保证不可变性，但也意味着每次 action 都会克隆整棵 Schema 树。

## 3. 数据模型（Schema 设计）

### 3.1 顶层结构

```
FormSchema
 ├─ schemaVersion: number        # 当前为 2
 ├─ title: string
 ├─ description?: string
 └─ rows: RowSchema[]
      └─ RowSchema
           ├─ id, type: "row"
           └─ columns: ColSchema[]
                └─ ColSchema
                     ├─ id, type: "col"
                     ├─ span: number        # 1-12，栅格宽度
                     └─ children: FieldSchema[]
```

### 3.2 字段类型（FieldType）

| 类型 | 说明 | 特有属性 |
|---|---|---|
| `input` | 单行输入 | placeholder, defaultValue |
| `textarea` | 多行输入 | placeholder, defaultValue, rows |
| `number` | 数字输入 | placeholder, defaultValue, min, max, step |
| `select` | 下拉选择 | placeholder, defaultValue, options |
| `checkbox` | 复选 | defaultValue(boolean) |
| `radio` | 单选 | defaultValue, options |
| `date` | 日期 | defaultValue(YYYY-MM-DD) |
| `divider` | 分割线 | 无 |
| `section` | 分组标题 | title, description |

所有字段共享 `BaseField`：`id / type / name / label? / helpText? / required? / disabled?`。

### 3.3 Schema 版本兼容

- `schemaVersion: 2`：当前标准结构（rows/columns/children）。
- v1 兼容：识别 `{ fields: [] }` 结构，自动迁移为「单行单列（span=12）」包裹所有字段。
- 无法识别的结构：**当前直接静默回退为空白默认表单**（见第 7 章「已知问题」P1-2）。

## 4. 功能规格

### 4.1 设计器模式（mode = "design"）

#### 4.1.1 三栏布局（FormDesigner.tsx）

| 区域 | 栅格占比 | 组件 | 功能 |
|---|---|---|---|
| 左栏 | 3/12 | ComponentPalette | 添加 Row / 添加各类型字段 |
| 中栏 | 6/12 | Canvas | 拖拽排序、选中 Row/Col/Field、增删列、增删行 |
| 右栏 | 3/12 | PropertyEditor | 编辑表单级/行级/列级/字段级属性 |

#### 4.1.2 组件库（ComponentPalette）

- **添加行容器**：受 `canLayout` 控制。
- **添加字段**（基础输入 / 选择类 / 结构类三组）：受 `canAddField` 控制。
- 添加字段的落位规则（优先级从高到低）：
  1. 若无任何行 → 自动创建默认行（7/5 两列）并将字段放入第一列；
  2. 若当前选中节点是 Col 或 Field → 插入该字段所在列；
  3. 兜底 → 插入第一行第一列。

#### 4.1.3 画布（Canvas）

- **行排序**：拖拽 RowCard，`DndContext` + `SortableContext`（`verticalListSortingStrategy`），松开后 dispatch `MOVE_ROW`。
- **列内字段排序 / 跨列移动**：拖拽过程中实时判定 `over` 目标（field 或 col），通过 `handleDragOver` 持续 dispatch `MOVE_FIELD`，实现"悬停即移动"的类 Notion 体验。
- **12 栅格布局**：每列通过 `col-span-{span}`（Tailwind，已 safelist 1–12）控制宽度。
- **空态提示**：0 行时显示引导文案；列内 0 字段时显示"把字段拖到这里"提示。
- **选中态**：点击 Row / Col / Field 时 `dispatch({ type: "SELECT", ... })`，驱动右侧属性面板联动。
- 所有拖拽/删除/新增交互均受 `usePermissions()` 返回的权限开关控制（`canReorder` / `canLayout` / `canDelete` / `canAddField`）。

#### 4.1.4 属性面板（PropertyEditor）

按当前 `selected` 节点类型展示不同面板：

| 选中类型 | 可编辑内容 |
|---|---|
| 表单（默认态） | 标题、描述 |
| Row | 添加/删除列、查看各列 span 与字段数、删除整行 |
| Col | 修改 span（1–12）、查看字段数、删除该列（至少保留 1 列） |
| Field | id/type（只读）、name、label、（分组类型的 title/description）、helpText、placeholder、rows（textarea）、min/max/step（number）、required、disabled、options（select/radio）、删除该字段 |

#### 4.1.5 顶部工具栏（TopBar）

- 显示 Schema 版本号、当前角色、Schema 校验状态（问题数 / OK）。
- **模式切换**：设计 ⇄ 预览（受 `canPreview` 控制）。
- **导入/导出面板**：文本框粘贴导入、JSON/YAML 格式切换、导出前展示实时校验错误列表（受 `canImportExport` 控制）。
- **从文件导入**：优先使用 `showOpenFilePicker`，浏览器不支持时提示改用面板粘贴。
- **导出为文件 / 导出到文件夹**：优先 `showSaveFilePicker` / `showDirectoryPicker`，均不支持时降级为 `<a download>` 浏览器下载。

### 4.2 预览/运行时模式（mode = "preview"）

- 由 `FormRuntime` 渲染，基于 Schema 生成**受控表单**（与设计态的 `FormRenderer` 只读渲染是两套独立实现）。
- 初始值规则：checkbox 默认 `false`，number 默认 `0`，其余默认 `""`，均可被字段 `defaultValue` 覆盖。
- **必填校验**：仅做前端简单非空校验（checkbox 要求 `=== true`，其余要求非空字符串/非 null），实时展示未通过的字段提示文案。
- **重置**：一键恢复为 Schema 初始值。
- **导出 values**：受 `canExportValues` 控制，展开只读 JSON 文本框展示当前填写值。
- Schema 变化时（如设计态切回后再切预览）自动通过 `useEffect` 重置 values。

### 4.3 权限矩阵

| 权限点 | superadmin | admin | ops | hr | sales | investigator | user |
|---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| canDesign | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| canLayout（增删行/列、改 span） | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| canEditProps（编辑字段属性） | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| canAddField | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| canDelete | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| canReorder（拖拽排序） | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| canImportExport | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ✅ |
| canPreview | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| canExportValues（导出填写结果） | ✅ | ✅ | ✅ | ✅ | ❌ | ✅ | ❌ |

> ⚠️ 规格声明与当前实现存在差距，详见第 7 章「已知问题」P0-2。规格要求：**所有列出的权限点必须在触发副作用（dispatch）前统一校验，UI 层禁用只是辅助，不能作为唯一防线**。

## 5. 关键交互流程

### 5.1 拖拽移动字段（跨列）流程

```
1. onDragStart：解析 active.id（field:<rowId>:<colId>:<fieldId>），
   记录来源 { rowId, colId, index } 到 activeFieldRef
2. onDragOver：解析 over.id
   ├─ over 是另一个 field → 计算 overIndex，dispatch MOVE_FIELD，更新 activeFieldRef
   └─ over 是一个 col 容器 → 插入到该列末尾，dispatch MOVE_FIELD，更新 activeFieldRef
3. onDragEnd：仅处理 Row 级排序（field 的落位已在 onDragOver 中完成），
   清空 activeFieldRef
```

**规格要求（当前实现有缺陷，见 P0-1）**：无论列内是否已有字段，列容器都必须能作为合法的 drop target 被 dnd-kit 识别，即必须通过 `useDroppable`（或等价 API）注册，而非仅设置 DOM `id` 属性。

### 5.2 Schema 导入流程

```
用户粘贴/选择文件
   → parseSchemaText（按扩展名或内容尝试 JSON→YAML）
   → dispatch SET_SCHEMA
   → designerReducer.migrateSchemaIfNeeded
        ├─ schemaVersion===2 且 rows 为数组 → 直接使用
        ├─ 存在 fields 数组（v1）→ 自动迁移为单行单列
        └─ 其他 → 规格要求：必须报错并阻止导入（当前实现回退为空白表单，属缺陷）
   → deepClone 写入 state.schema
   → selected 重置为表单级，mode 重置为设计态
```

### 5.3 Schema 导出流程

```
serializeSchema(format) → JSON.stringify / YAML.dump
   → 优先系统保存对话框（File System Access API）
   → 不支持时降级为浏览器下载（Blob + <a download>）
```

## 6. 非功能需求

| 类别 | 要求 |
|---|---|
| 兼容性 | File System Access API 为渐进增强，必须在不支持的浏览器（Firefox/Safari 等）中降级为下载/粘贴导入，不能报错阻断 |
| 安全性 | YAML 解析必须使用安全 schema（禁止 `!!js/function` 等自定义 tag 反序列化），当前 js-yaml 4 的 `load()` 默认安全，需在后续升级中维持该约束 |
| 性能 | 单表单字段数 ≥ 100 时，属性面板输入操作（含标题/描述/字段属性文本框）响应延迟应 < 100ms；当前实现每次按键都会深拷贝整棵 Schema 并触发全树重渲染，需优化（见 P2-1） |
| 数据一致性 | 任意 reducer action 执行后，Schema 必须始终满足：每行至少 1 列、每个字段 name 在校验前允许重复但需高亮提示、col span 之和建议 ≤ 12 |
| 权限一致性 | 同一权限点（如 canLayout / canDelete / canEditProps）在设计器的任意入口（画布 / 属性面板 / 组件库）都必须表现一致，不允许"画布禁用但属性面板未禁用"的绕过路径 |
| 可维护性 | 设计态字段渲染与运行态字段渲染应收敛为同一套底层组件，避免 UI 变更需要双处同步修改 |

## 7. 已知问题与待办清单（按优先级）

### P0（阻断性 / 安全性，需优先修复）

1. **空列无法作为拖拽落点**：`Canvas.tsx` 中列容器仅设置了 DOM `id` 属性，未通过 `useDroppable` 注册为 dnd-kit 可放置区域，导致向空列拖入字段大概率不生效。
2. **权限校验未下沉到状态层，且 UI 层存在遗漏**：
   - `designerReducer` 对所有 action 无条件执行，理论上可被绕过 UI 直接触发越权操作；
   - `PropertyEditor.tsx` 的 Row 面板「+ 添加列」「删除该行」按钮缺少权限判断；
   - Field 面板中 `name / label / title / description / helpText / placeholder / rows / min/max/step / required / disabled` 等属性编辑均未做 `canEditProps` 判断；
   - 表单标题/描述的权限判断写在 `onChange` 内部（静默不生效），未同步 `disabled` 输入框，体验和安全性都不达标。

### P1（数据健壮性）

1. `DELETE_COL` 在 reducer 层缺少"至少保留一列"的兜底，可能导致 `ComponentPalette.addField` 兜底逻辑访问越界抛异常。
2. Schema 导入失败（既非 v2 也非 v1 `fields` 结构）时静默回退为空白表单，用户数据被无提示吞掉。
3. number 类型字段在清空输入框时被写入空字符串 `""`，与声明类型 `number` 不一致。

### P2（性能与可维护性）

1. 每次属性编辑（含逐字符输入）都触发整棵 Schema `structuredClone` + 全树重渲染，未做防抖/局部 state 缓冲，也未对 `RowCard` / `FieldCard` 做 `memo`。
2. `fieldRenderers.tsx`（`FieldView`）与 `FormRuntime.tsx`（`RuntimeField`）渲染逻辑高度重复，且 `FormRenderer.tsx` 在现有引用关系中疑似未被使用（待确认是否为死代码）。
3. `PropertyEditor.tsx` 中大量使用 `as any` 类型断言，弱化了 `FieldSchema` 联合类型带来的编译期保护。

### P3（体验优化）

1. 当前角色通过硬编码写死在 `App.tsx` 源码中，建议提供开发态角色切换器便于联调测试。
2. 字段 `name` 唯一性仅能通过导入导出面板的校验错误列表延迟发现，属性面板编辑时无实时重名提示。
